# 数据仓库

为企业决策提供数据支撑

存储、分析、计算

查看日志的方法：打开控制台管理，network输入log

  • 然后搜索url解析，复制的log数据进行解码

数据的来源：

  • 业务数据：mysql等
  • 用户行为数据：埋点：日志服务器(文件)
  • 爬虫数据

数据仓库：Hive on spark

ODS备份=>DWD清洗=>DWS(按天聚合)=>DWT(按年聚合）=>ADS(报表)

调度：Azkaban、Oozie、

数据传输：flume(日志)（大的话可用kafka消峰）、sqoop(数据库)

数据输出：报表系统、用户画像（统计，规则，机器学习）、推荐系统、机器学习

需求

  • 用户行为数据采集搭建
  • 业务数据采集搭建
  • 数据仓库维度建模
  • 分析、设备、会员、商品、地区、活动等核心主题报表，近100个
  • 采用即席查询工具对指标进行分析
  • 对集群性能进行监控，报警。
  • 元数据管理。
  • 质量监控
  • 权限管理

架构设计：

技术选型

因素：数据量大小、业务需求、行业内经验、技术成熟度、开发维护成本、总成本预算

采集传输：flume、kafka、sqoop、Logstash、DataX

数据存储：mysql、hdfs、hbase、redis、MongoDM

数据计算：Hive、Tez、Spark、Flink、Storm

数据查询：Presto、Kylin、Impala、Druid、ClickHouse、Doris

数据可视化：Echarts、SuperSet、QuickBi、Datav

调度工具：Azkaban、Oozie、DolphinScheduler、Airflow

集群监控：Zabbix(离线)、prometheus（实时）

权限管理：Ranger、Sentry

元数据管理：Atlas

  • 技术
  • 框架：apache、cdh、hdp
  • 服务器：物理、云主机
  • 集群规模

集群分俩套：生产or部署

生产集群：10台(看需求)

  • 消耗资源分开
  • 数据传输紧密防止一起
  • 客户端尽量放在一到俩台服务器上，方便外部访问
  • 有依赖关系的尽量放到同一台服务器。

# 同步策略

  • 全量表：完整的 （适 数据量小，可会发生变化）

    每日全量表： 按日期分区

    每日存储一次作为一个分区

  • 增量表：新增加的 （适 数据量大、只新增不变化）

    按日期分区，保存业务系统新增数据

  • 新增变化表：增加和变化的 （ 量大+变化）

    业务数据库新增加的数据和未发生变化的进行整合

  • 特殊表：只存一次 （不变化）

    不发生变化的表，地区，省份，民族等。

第一次同步时要采用全量+对应的方式

#! /bin/bash

APP=gmall
sqoop=/opt/module/sqoop/bin/sqoop

if [ -n "$2" ] ;then
   do_date=$2
else 
   echo "请传入日期参数"
   exit
fi 

import_data(){
$sqoop import \
--connect jdbc:mysql://node1:3306/$APP \
--username root \
--password hadoop \
--target-dir /origin_data/$APP/db/$1/$do_date \
--delete-target-dir \
--query "$2 where \$CONDITIONS" \
--num-mappers 1 \
--fields-terminated-by '\t' \
--compress \
--compression-codec lzop \
--null-string '\\N' \
--null-non-string '\\N'

hadoop jar /opt/module/hadoop-3.1.3/share/hadoop/common/hadoop-lzo-0.4.20.jar com.hadoop.compression.lzo.DistributedLzoIndexer /origin_data/$APP/db/$1/$do_date
}

# 数仓分层

ODS(原)=> (DWD(细)=>DWS(汇)=>DWT(题))(DIM 维度)=>ADS(应)

  • ODS：存放加载原始数据，不做处理

  • DWD: 对ODS的数据进行清洗(去除空值，过滤脏数据，超出范围的数据，脱敏)等 一行一次行为

  • DIM:维度数据，对业务的描述信息，何人，何时、何地等，

    • DWS 对DWD按每天，进行轻度汇总。 一行一天的行为

    • DWT :对DWS按数据累加汇总 一行一个对象的行为累积行为 :最近n天

  • ADS: 为各种统计报表提供数据

分层的意义

  • 把复杂问题简单化
  • 减少重复开发
  • 隔离原始数据

数仓和数集的区别：

  • 数仓：为整个企业部门提供决策

  • 数据集市：部门级，微型数据仓库

    独立形：(数据来自业务)

    从属形：(数据来自数据仓库)

数仓命名规范：

  • 表命名：对应层级的表加上其前缀，例如特殊表 temp_ 前缀命名，DWT层用 dwt_命名

  • 脚本命名：数据源_to_目标_db/log.sh ： 行为以log为后缀，业务以db为后缀

  • 表字段：

    数量 bigint、金额decimal(16,2) 16位，2位小数

    字符串：(名字，描述)、主键外键

    时间戳：bigint

数仓理论：

  • 关系建模 (业务系统)

    范式：降低数据的庸余性，保证数据的一致性

    • 拆分多，jion多
    • 1范式=>2范式=>3范式=>4范式=>5范式

    属性不可切=>不存在非主键部分依赖

    函数依赖：完全(直接)、部分(间接)、传递(联系)

    表多

  • 维度建模 (普遍，重点学习)

    • 事实表(具体的业务)

      非常大，内容相对窄，列数少，经常发生变化

      每行代表一个业务事件（下单支付等）

      分类

      • 事务型 ：每个事件为单位，不发生变化,增量同步
      • 周期型快照：保存固定时间间隔的数据 只关心结果，全量同步
      • 累积型快照：跟踪业务事实变化，会发生周期性变化的业务，新增+变化

    • 维度表(业务的描述信息)

      表宽，行数相对较小，内容相对固定（编码表）

      例：地区等

      分类：

      • 星型模型：性能优先

      • 雪花模型：灵活优先

        (会对维度表进行规范化)

      • 星座模型（多个星型模型交着在一起）

    查询简单效率高

但是Kylin所依赖的HBase需要进行额外的配置，才能在安全的Hadoop环境下正常工作。

  • HBase开启Kerberos认证
  • Kylin进行Kerberos认证

# 数据质量检测

对数仓各个阶段的数据质量，进行识别检测，利用python+shell进行识别，存储入库，然后再用superset报表

指标：

*评价标准* *描述* *监控项*
*唯一性* 指主键保持唯一 字段唯一性检查
*完整性* 主要包括记录缺失和字段值缺失等方面 字段枚举值检查
字段记录数检查
字段空值检查
*精确度* 数据生成的正确性，数据在整个链路流转的正确性 波动阀值检查
*合法性* 主要包括格式、类型、域值的合法性 字段日期格式检查
字段长度检查
字段值域检查
*时效性* 主要包括数据处理的时效性 批处理是否按时完成
