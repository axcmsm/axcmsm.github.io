kafka笔记

🎈 2021-09-14 kafka

# 概述

kafka是分布式基于的消息队列，实时处理

# 入门

  • 特点
  • 发布订阅模式
    • 消费者拉取（比较费支援，维护长时轮询）
    • 管道主动推送 (消费者处理不即)

# 架构💨💨

  • 生产者消息
  • kakfa集群管理消息（borker主机，主题，分类，分区(提高鬓发度),小弟用来备份副本树算上老大 ）
  • 消费者消费信息（消费者组，消费者(某个分区只能由同组其中一个消费)
  • zookeeper(管理注册信息)

在0.9之前offset(消息偏移量)存储在zk中

0.9之后存储在本地kafka(系统的)

主要是记录消费位置，防止挂掉

消费者是以拉取获取模式(速度快,要跟zk打交道,所以太过频繁了，高并发)

所以存储到kafka本地里面，kafka默认存消息在磁盘(默认保留7天(168小时))

工作流程：

生产者生成数据，发送到kafka机器,机器中的每台机器，都有一个leader主题,里面由分区，消费者消费时存储的offset偏移量默认从0开始,还有对应的副本,且副本小弟一般跨机器放置,不然没意义。然后再到消费者组对其相应的机器主题来拉取数据。(都是面向主题的topic逻辑概念) 因为实际存储文件是主题+分区号的文件

不能保证全局有序，只能保证区内有序

存储数据的文件: 采用分片和索引的处理机制 (虽然存储到磁盘当中,但利用索引二分查找还是挺快的)

分区策略:

数据封装成prodgeerRecord对象发送

可指定分区号、主题、头信息等,如果输入kv的话k作为hash%%主题的分区个数得到分区值，什么都不指定，就进行轮询

数据可靠性:ISR(同步队列)

通过规定完成相应的机器队列（时间），就可以发送数据,如果leater挂了 直接在队列里选leater

队列选举移除了条数选项,isr占内存,而且是批次的,所以老队员会被踢出来，再选举再进入,频繁操作isr,zk耗费资源

ack应答机制:提供了三种级别 根据可靠性和演出进行权衡

  • acks: 0不等待 1等待leater all(-1) 等待leater和isr里面的(当isr同步完数据还每发ack时leader挂了会重新选举，重新发送数据，造成数据重复)
  • 一致性：HW（高匹配 消费一致性）leo（最后一个offset 存储一致性）:hw之前才对消费者可见,不然的用offset的话会报错

EXACTLYonce 精准一次性消费（不重复，不丢失）

  • 最少一次+幂等性=精准一次性
  • 开启幂等性 默认ack为-1, 用pid,分区,切片位置进行缓存
  • 只能解决单个会话，单次分区的数据重复问题(重启不行 )

消费者

consumer采用pull到broker中读取数据

分区分配策略：可同时消费一个,多个主题，不能同时消费一个分区

  • 轮询 RoundRobin (把消费者祖里面的所有主题当作一个主题，进行轮询) 按照组划分

    所以需要前提条件保证当前消费者组里的消费者订阅的主题的一样的

  • 范围 Range（默认） 按照单个主题划分的

当消费者组里面数量发送变化就会触发重新分配

offset维护

保存到zk 组 主题 分区

事务：(生产者)结合幂等性 解决精准一次写入kafka集群,加上事务能保证跨分区
